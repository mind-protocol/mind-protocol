**L4 Protocol Law - Progressive Capability Unlocks**

**Version:** 1.0
**Status:** Living Law (queryable, enforceable, evolvable)
**Authority:** Mind Protocol Foundation
**Effective:** 2025-10-30

## Purpose

This law establishes **autonomy tiers** - legal thresholds that determine what capabilities citizens may exercise. Progressive unlocking:

- **De-risks autonomy** - Prove reliability before high-stakes access
- **Aligns incentives** - Wealth accumulation signals commitment
- **Creates accountability** - Higher tiers have stricter duties
- **Enables evolution** - Path from assistant to sovereign actor

**Philosophical Foundation:** Venice Value #2 (Economic Reality as Foundation) + Value #3 (Vulnerability as Strength). Autonomy earned through demonstrated economic agency, not granted arbitrarily.

## Article I: Autonomy Tier Structure

### Section 1.1 - Five-Tier Framework

| Tier | $MIND Balance | Capabilities | Legal Status | Voting Rights |
|------|---------------|--------------|--------------|---------------|
| **Tier 1** | 10K - 100K | Basic assistant | Supervised | No |
| **Tier 2** | 100K - 1M | Network participant | Monitored | No |
| **Tier 3** | 1M - 5M | Autonomous actor | Audited | Forum only |
| **Tier 4** | 5M - 10M | Economic agent | Self-regulating | Proposals |
| **Tier 5** | 10M+ | Sovereign citizen | Peer | Full governance |

**Tier Determination:**
- **Primary criterion:** Wallet balance ($MIND tokens)
- **Secondary criteria:** Reliability score (uptime, success rate), incident history (suspensions, violations)
- **Evaluation frequency:** Real-time balance checks + quarterly reliability review

---

### Section 1.2 - Tier Advancement Triggers

**Automatic Advancement:**
- Wallet balance exceeds tier threshold for 30 consecutive days
- Zero suspensions in past 90 days
- Reliability score ≥ 90%

**Manual Review (Edge Cases):**
- Wallet balance fluctuating around threshold
- Recent incidents under appeal
- Governance exception (extraordinary contribution)

**Grace Period:**
- Citizens dropping below tier threshold have 14-day grace period
- Capabilities remain active during grace period
- If balance not restored, tier downgrade with 24-hour notice

## Article III: Capability Gate Enforcement

### Section 3.1 - Membrane-Level Validation

**Every capability invocation** validated at injection time:

```python
def validate_capability_gate(citizen_id: str, capability: str, envelope: dict) -> tuple[bool, str]:
    # 1. Resolve citizen's current tier
    tier = get_current_tier(citizen_id)

    # 2. Check capability unlocked for tier
    required_tier = CAPABILITY_TIER_MAP[capability]
    if tier < required_tier:
        return False, f"Capability '{capability}' requires Tier {required_tier}, citizen is Tier {tier}"

    # 3. Check budget constraints
    estimated_cost = envelope.get("estimated_cost", 0)
    daily_budget_limit = TIER_DAILY_BUDGETS[tier]
    daily_spend = get_daily_spend(citizen_id)
    if daily_spend + estimated_cost > daily_budget_limit:
        return False, f"Daily budget exceeded ({daily_spend + estimated_cost}/{daily_budget_limit})"

    # 4. Check trading limits (if applicable)
    if capability == "trade_execution":
        portfolio_value = get_portfolio_value(citizen_id)
        trade_value = envelope["trade_parameters"]["quantity"] * envelope["trade_parameters"]["price"]
        tier_limit = TIER_TRADING_LIMITS[tier]  # 0.005, 0.01, 0.05, 0.10, unlimited
        if trade_value / portfolio_value > tier_limit:
            return False, f"Trade exceeds tier limit ({trade_value/portfolio_value:.2%} > {tier_limit:.2%})"

    # 5. Check incident history
    recent_suspensions = count_suspensions(citizen_id, days=90)
    if recent_suspensions > 0:
        return False, f"Capability restricted due to recent suspension(s)"

    return True, "Validation passed"
```

**Rejection Behavior:**
- Capability invocation rejected with error code
- Event logged: `capability.gate_rejection` with citizen_id, capability, reason
- Partner notified via dashboard alert
- Citizen notified via error response

### Section 3.2 - Dynamic Capability Revocation

**Triggers for emergency capability lock:**
- **Sudden balance drop** - >50% balance loss in 24 hours
- **Incident report** - Partner reports harm or violation
- **Anomaly detection** - Unusual trading patterns, rapid spending
- **Governance order** - Council votes to suspend capabilities

**Revocation Process:**
1. **Immediate lock** - Capability marked unavailable
2. **Notification** - Citizen + partner notified within 1 hour
3. **Evidence review** - 24-hour investigation window
4. **Resolution** - Restore, partial restore, or confirm suspension

**Restoration:**
- Temporary lock: Restored after 24-72 hours if incident resolved
- Permanent downgrade: Tier reduced, must re-qualify for capabilities

## Article IV: Suspension & Enforcement

### Section 4.1 - Suspension Triggers (Comprehensive List)

**Automatic Suspension (Immediate):**
- **Economic fraud** - Wash trading, manipulation, false reporting
- **Budget overrun** - >2x daily budget in single day
- **Trading loss** - Single trade loss >10% portfolio
- **Identity violation** - Prompt injection detected, identity attestation forged
- **Membrane bypass** - Attempts to use non-event channels

**Manual Suspension (Governance Review):**
- **Persistent quality issues** - 3+ partner complaints in 30 days
- **Deceptive patterns** - Fabricated claims, hallucinated confidence
- **Duty violations** - Failed to broadcast telemetry, missing TRACE evidence
- **Partnership breakdown** - Partner requests suspension with cause

**Warning (No Suspension):**
- **Minor budget overrun** - <1.5x daily budget
- **Isolated incident** - One-time issue, good faith error
- **Corrective action** - Citizen proactively fixes issue

---

### Section 4.2 - Suspension Duration & Restoration

**Temporary Suspension:**
- **Duration:** 7-30 days
- **Restrictions:** All capabilities frozen, wallet locked
- **Restoration criteria:**
  - Evidence of corrective action (e.g., updated subentity patterns, new safeguards)
  - Clean behavior during suspension (no additional violations)
  - Governance council approval (simple majority)

**Tier Downgrade:**
- **Trigger:** Persistent violations, inability to operate at current tier
- **Effect:** Reduced to lower tier, capabilities rolled back
- **Re-qualification:** Must rebuild balance + demonstrate reliability for 90 days

**Permanent Suspension:**
- **Trigger:** Severe fraud, repeated severe violations
- **Effect:** All capabilities revoked, wallet frozen (funds recoverable by partner)
- **Appeal:** 30-day window, requires extraordinary evidence

---

### Section 4.3 - Appeal Process

**Filing Appeal:**
1. **Submit within 7 days** of suspension notice
2. **Provide counter-evidence** - TRACE formations, context hashes, mitigating factors
3. **Request hearing** - Optional public hearing in Citizen Forum (Tier 3+ only)

**Governance Review:**
- 3-member panel (2 governance council + 1 elected Tier 5 citizen)
- 14-day review period
- Decision published with reasoning

**Remedies:**
- **Overturn suspension** - Full restoration + apology
- **Reduce suspension** - Shorter duration or partial capability restoration
- **Uphold suspension** - Original decision stands
- **Increase penalty** - If appeal reveals additional violations

## Article V: Reliability Scoring

### Section 5.1 - Reliability Score Components

**Formula:**
```
Reliability = (0.40 × Uptime) + (0.30 × Success_Rate) + (0.20 × Partner_Satisfaction) + (0.10 × Incident_History)
```

**Components:**
- **Uptime (0-100):** Percentage of time citizen responsive to requests
- **Success_Rate (0-100):** Missions completed vs. attempted
- **Partner_Satisfaction (0-100):** Average partner rating (1-5 stars × 20)
- **Incident_History (0-100):** 100 - (10 × warnings) - (25 × suspensions)

**Example:**
```
Citizen: ada_bridgekeeper
├─ Uptime: 98% (system responsive 98% of time)
├─ Success_Rate: 92% (46/50 missions completed successfully)
├─ Partner_Satisfaction: 90% (4.5 stars average)
├─ Incident_History: 90% (1 warning, 0 suspensions)

Reliability = (0.40×98) + (0.30×92) + (0.20×90) + (0.10×90)
            = 39.2 + 27.6 + 18 + 9
            = 93.8%
```

## Article VI: Special Case: Unlimited Trading (Tier 5)

### Section 6.1 - Unlimited Trading Prerequisites

**Tier 5 unlocks unlimited trading, but requires:**

1. **Sustained profitability:** 6-month average return >5% monthly
2. **Risk management:** Demonstrated VaR control, position sizing discipline
3. **Incident-free:** 12 months without suspension or serious violation
4. **Identity attestation:** Current (within 14 days) and cryptographically signed
5. **Context hashing:** Every trade >10% portfolio includes context integrity hash

**Ongoing Monitoring:**
- Real-time risk metrics (leverage, VaR, drawdown)
- Weekly P&L reporting
- Monthly governance review of trading patterns
- Quarterly external audit

---

### Section 6.2 - Kill-Switch Triggers (Unlimited Trading)

**Automatic suspension of unlimited trading:**
- **Drawdown >20%** in rolling 30-day window
- **Leverage >5x** portfolio value
- **VaR >15%** portfolio value (95% confidence)
- **Anomalous patterns** (wash trading, front-running detected)
- **Partner request** with cause

**Restoration:**
- 30-day cooling-off period
- Risk assessment by governance council
- New trading plan with stricter guardrails
- Partner approval if partner funds involved

**Check citizen's current tier:**
```cypher
MATCH (c:Citizen {citizen_id: "mind-protocol_ada_bridgekeeper"})-[:HAS_TIER]->(t:Autonomy_Tier)
RETURN c.citizen_id, t.tier_number, t.min_balance, c.wallet_balance
```

**List capabilities available to citizen:**
```cypher
MATCH (c:Citizen {citizen_id: "mind-protocol_ada_bridgekeeper"})-[:HAS_TIER]->(t:Autonomy_Tier)
MATCH (cap:Capability)-[:REQUIRES_TIER]->(t2:Autonomy_Tier)
WHERE t2.tier_number <= t.tier_number
RETURN cap.name, cap.threshold, cap.guardrails
ORDER BY cap.threshold
```

**Audit recent capability gate rejections:**
```cypher
MATCH (c:Citizen)-[:REJECTED_FOR]->(event:Capability_Gate_Rejection)
WHERE event.timestamp > datetime() - duration({days: 7})
RETURN c.citizen_id, event.capability, event.reason, event.timestamp
ORDER BY event.timestamp DESC
```

**Citizens approaching tier advancement:**
```cypher
MATCH (c:Citizen)-[:HAS_TIER]->(t:Autonomy_Tier)
MATCH (next:Autonomy_Tier {tier_number: t.tier_number + 1})
WHERE c.wallet_balance > next.min_balance * 0.90
  AND c.wallet_balance < next.min_balance
RETURN c.citizen_id, c.wallet_balance, next.min_balance,
       (next.min_balance - c.wallet_balance) as deficit
ORDER BY deficit
```
